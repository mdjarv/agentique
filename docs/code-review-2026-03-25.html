Cross-Cutting Themes
Disconnected State Systems
Frontend hook local state vs Zustand store vs backend broadcasts. The rebase conflict bug we just fixed is one instance of a systemic pattern. Store mutations scattered across lib/, hooks/, components/.
Session Package God Object
Backend session/ handles lifecycle, git ops, event streaming, state machine, broadcasting, approval callbacks — 800+ lines in session.go alone. GitService mixes git orchestration with broadcasting and Haiku generation.
Concurrency Gaps
WS write loop race conditions, unprotected seqInTurn field, TOCTOU races in Manager.Get(), broadcast-during-lock deadlocks. Multiple lock/unlock cycles in Session.Query().
No Interfaces = Untestable
Backend: Manager, store.Queries, gitops all concrete. Frontend: global WS singleton, store.getState() called directly in libs/components. Neither layer can be unit tested in isolation.
Silent Failures Everywhere
Backend ignores DB update errors (10+ underscore assignments). Frontend swallows promise rejections with console.error. No error boundaries. Result types mix error/success ambiguously.

P0 Detail: WS Write Loop Race
In the writeLoop drain section (when ctx.Done()), the mutex is unlocked between SetWriteDeadline and WriteJSON. Another goroutine could write between these operations. Additionally, the regular message case and ping case each lock/unlock independently — if context cancels during a regular write, both branches execute concurrently. Fix: Hold the mutex for the entire write operation. Consider a single writer goroutine pattern.
P0 Detail: Broadcast Deadlock
broadcastState() acquires mu, reads state, then calls s.broadcast() which enqueues to the WS send channel. If the channel is full, conn.send() closes the connection — which may trigger cleanup that tries to acquire the same lock. Fix: Collect broadcast data under the lock, then release the lock before calling broadcast.
P0 Detail: Orphaned CLI Processes
Manager.Create() uses context.Background() for client.Connect() so the CLI process outlives the WS connection. If the server crashes or is killed, the background context is never cancelled — Claude CLI processes become orphans. Fix: Track PIDs. On startup, check for orphaned processes from previous runs. Use a process group or SIGCHLD handler.
